2. Tarea 1 — Modelado de Densidad de Vulnerabilidades

La densidad de vulnerabilidades Vd cuantifica el riesgo del software en función de su tamaño:

Vd = Σ(vi × wi) / (L / 1000)
wi: Baja = 0.1  |  Media = 0.5  |  Alta = 1.0  |  Umbral aceptable: Vd < 2.0

5.3 Justificación del Falso Positivo — bcryptjs
FALSO POSITIVO Hallazgo: Snyk puede marcar bcryptjs@2.4.3 por considerarlo "potencialmente inseguro".

Justificación: Este hallazgo es un falso positivo porque:
  • El código utiliza bcrypt.hash(password, 10) con factor de costo 10, valor mínimo recomendado por OWASP para producción.
  • bcryptjs@2.4.3 no registra CVEs activos en las bases de datos NVD ni OSV.
  • Prisma ORM elimina la posibilidad de inyección SQL en todas las consultas al usar prepared statements internamente.
  • Snyk analiza el patrón de uso de la librería sin contexto de los parámetros reales, generando una alerta conservadora que no representa un riesgo real en este proyecto.

6. Preguntas de Control
1. ¿Cuál es la diferencia fundamental entre el análisis SAST (Static) y el análisis DAST (Dynamic)?
El análisis SAST examina el código fuente, bytecode o binario sin ejecutar la aplicación, identificando patrones de código inseguro en tiempo de desarrollo. El análisis DAST ataca la aplicación en ejecución enviando entradas maliciosas a los endpoints activos y observando las respuestas. En NutriHealth Pro, Snyk aplicó SAST analizando los archivos .ts directamente; un DAST enviaría payloads al endpoint /api/auth/login buscando respuestas anómalas. SAST detecta problemas más temprano en el ciclo pero genera más falsos positivos; DAST tiene menos falsos positivos pero requiere un ambiente de ejecución funcional.
2. ¿Por qué el análisis SAST suele generar una mayor tasa de falsos positivos en comparación con las pruebas de penetración manuales?
SAST analiza todos los caminos de ejecución posibles del código de forma estática, sin conocer el contexto real de ejecución ni los datos de entrada reales. Por ejemplo, Snyk marcó bcryptjs porque el análisis abstracto no puede verificar que el parámetro saltRounds=10 cumple el estándar OWASP. Una prueba de penetración manual verificaría empíricamente que el hash producido es computacionalmente inviable de romper. SAST aplica reglas conservadoras para no omitir ninguna vulnerabilidad, lo que incrementa los falsos positivos como costo de la cobertura total.
3. Defina el concepto de "Análisis de Composición de Software" (SCA) y su importancia en la cadena de suministro.
El SCA (Software Composition Analysis) identifica y audita todas las dependencias de terceros de un proyecto contra bases de datos de vulnerabilidades conocidas (CVE/NVD). En NutriHealth Pro, el comando snyk test analizó 6 dependencias directas del backend: express, jsonwebtoken, bcryptjs, axios, openai y prisma. Su importancia en la cadena de suministro radica en que un atacante puede comprometer el sistema a través de una librería vulnerable aunque el código propio sea seguro (ataque de tipo supply chain, como el incidente SolarWinds 2020), sin que el desarrollador lo detecte durante la revisión de código ordinaria.
4. ¿Cómo ayuda la integración de SAST en un pipeline de CI/CD a reducir el "Costo de Remediación"?
El modelo de costo relativo de remediación (IBM Systems Sciences Institute) establece que corregir un defecto en producción cuesta entre 15× y 100× más que en la fase de diseño. Integrar Snyk en GitHub Actions para NutriHealth Pro permitiría bloquear automáticamente cualquier Pull Request que introduzca nuevas vulnerabilidades medias o altas, como la divulgación de errores en aiController.ts, antes de que llegue a la rama principal. Esto garantiza que los hallazgos se remedien en el contexto del desarrollador que escribió el código, cuando el costo de corrección es mínimo, en lugar de descubrirse en auditorías post-despliegue.
5. Explique el principio de "Seguridad desde la Izquierda" (Shift Left Security).
Shift Left Security consiste en desplazar las actividades de seguridad hacia las fases tempranas del ciclo de vida del desarrollo de software (SDLC), en lugar de aplicarlas únicamente antes del despliegue. En NutriHealth Pro esto se manifestaría instalando SonarLint en VS Code para detectar el uso de any mientras el desarrollador escribe el código, ejecutando Snyk en cada commit mediante pre-commit hooks, y revisando la arquitectura de seguridad (ausencia de rate limiting) en la fase de diseño. El resultado es que los defectos de seguridad se corrigen en minutos durante el desarrollo, en lugar de en días o semanas durante una auditoría posterior.
7. Conclusiones
  1. Se ejecutó exitosamente el análisis SAST sobre el proyecto NutriHealth Pro utilizando Snyk CLI en Kali Linux, identificando 5 hallazgos (0 altos, 2 medios, 3 bajos), lo que confirma que la aplicación de herramientas automatizadas de análisis estático es viable y efectiva incluso en proyectos de mediana escala desarrollados en TypeScript.
  2. La densidad de vulnerabilidades calculada fue Vd = 0.346, significativamente por debajo del umbral aceptable de 2.0, lo que demuestra que la arquitectura orientada a ORM (Prisma) y el uso de librerías de autenticación estándar (bcryptjs, jsonwebtoken) reducen de forma inherente la superficie de ataque frente a vulnerabilidades clásicas como inyección SQL y gestión insegura de contraseñas.
  3. La diferenciación entre vulnerabilidades reales y falsos positivos resultó esencial: el hallazgo sobre bcryptjs fue clasificado como falso positivo tras verificar la correcta implementación con factor de costo 10, evidenciando que los reportes SAST automatizados requieren análisis crítico del desarrollador para evitar remediar problemas inexistentes con impacto negativo sobre la funcionalidad.
  4. La vulnerabilidad más significativa detectada fue la divulgación de información en errores (CWE-209) en aiController.ts, cuya remediación es simple pero de alto impacto: exponer mensajes de error internos al cliente facilita el reconocimiento del sistema por parte de atacantes, violando el principio de mínima exposición de información.
  5. La práctica evidenció que la ausencia de rate limiting en endpoints de autenticación (CWE-307) es un error común en fases iniciales de desarrollo que SAST puede detectar preventivamente, y cuya inclusión en CI/CD habría bloqueado el merge de ese código antes de llegar a producción.
8. Recomendaciones
  • Integrar Snyk en el pipeline de GitHub Actions: configurar el workflow .github/workflows/security.yml para ejecutar snyk code test y snyk test en cada Pull Request, bloqueando merges con severidad media o superior, aplicando el principio Shift Left.
  • Implementar rate limiting con express-rate-limit: proteger todos los endpoints de autenticación (/api/auth/login) con un máximo de 10 intentos por IP cada 15 minutos, mitigando ataques de fuerza bruta sobre credenciales de nutricionistas y administradores.
  • Gestionar secretos con variables de entorno validadas al inicio: reemplazar los non-null assertions (!) por validación explícita de JWT_SECRET y OPENAI_API_KEY en el arranque del servidor, lanzando un error descriptivo que evite iniciar la aplicación en estado inseguro.
  • Implementar manejo centralizado de errores: unificar todas las respuestas de error 500 en el middleware errorHandler.ts para garantizar que ningún mensaje de excepción interna llegue al cliente, registrando los detalles únicamente en los logs del servidor con un identificador de correlación.
  • Ejecutar análisis SAST periódicamente (no solo en CI/CD): programar una ejecución semanal de snyk monitor para detectar nuevas vulnerabilidades en dependencias que aparezcan en bases de datos CVE después del último build, dado que el ecosistema npm publica actualizaciones de seguridad continuamente.
